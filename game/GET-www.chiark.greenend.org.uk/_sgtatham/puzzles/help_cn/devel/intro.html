第1章：简介

Puzzles 代码库分为四个部分：一组可互换的前端，一组可互换的后端，一个作为两者之间缓冲区的通用“中端”，以及一堆杂项实用函数。在以下各节中，我将对每个部分进行一些一般性讨论。

1.1 前端

前端是代码的非可移植部分：当你移植到不同平台时，你会完全替换这部分。因此，它负责所有系统调用、所有 GUI 交互以及其他任何平台特定的内容。

前端包含 main() 或本地平台的等效函数。应用程序执行流程的顶层控制属于前端（例如，它不是由其他地方的通用 main() 调用的一组函数）。

前端完全自由地为 Puzzles 的任何给定端口设计 GUI。例如，没有集中机制来维护菜单布局。这在一致性方面有成本（当我确实希望在多个平台上使用相同的菜单布局时，每次更改时我都必须并行编辑 N 段代码），但优点是可以遵循本地 GUI 约定并适应本地约束。例如，MacOS 有严格的人机界面指南，规定了与我在 Windows 和 GTK 上使用的不同的菜单布局；没有什么阻止 MacOS 前端提供与这些指南一致的菜单布局。

虽然前端在与代码其他部分的交互中主要是调用者而不是被调用者，但它需要实现一个小的 API 供其他模块调用，主要是游戏绘制图形时使用的绘制函数。绘图 API 在 第3章 中有文档说明；其他杂项前端 API 函数在 第4.40节 中有文档说明。

1.2 后端

在这个集合中，“后端”与“谜题”同义。每个后端实现不同的游戏。

在顶层，后端只是一个数据结构，包含一些常量（标志字、首选像素大小）和大量函数指针。后端几乎总是被调用者而不是调用者，这意味着后端主动执行操作的能力有限。

后端中的持久状态分为多个数据结构，这些数据结构用于不同的目的，因此可能会被代码的其余部分切换、更改且不通知，以及以其他方式更新。设计后端时，将正确的数据片段放入正确的结构中非常重要，否则中端提供的标准功能（如撤销）可能无法工作。

后端数据结构中提供的函数和变量在 第2章 中有文档说明。

1.3 中端

Puzzles 有一个单一且通用的“中端”。此代码对所有平台和所有游戏通用；它位于前端和后端之间，为所有地方提供标准功能。

添加新后端或新前端的人通常不需要编辑中端。在极少数情况下，可能需要对中端进行更改，以允许新游戏做当前中端设计未预期的事情；然而，这非常容易出错，在没有咨询主要维护者（我）的情况下可能不应该进行。包含未宣布的中端更改的补丁提交将像任何其他补丁一样根据其优点进行处理；这只是一个友好的警告，中端更改需要相当多的优点才能被接受。

中端提供的功能包括：

  • 维护游戏状态结构列表并在该列表上来回移动以提供撤销和重做。
  • 处理计时器（用于移动动画、完成时的闪烁，在某些情况下实际计时游戏）。
  • 处理游戏 ID 的容器格式：接收它们，将它们分解为参数、描述和/或随机种子等。游戏后端只需处理游戏 ID 的各个部分（编码参数和编码游戏描述）；其他所有内容由中端集中处理。
  • 处理标准按键和菜单命令，如“新游戏”、“重新开始游戏”和“退出”。
  • 预处理鼠标事件，以便游戏后端可以依赖它们以合理的顺序到达（没有缺少的按钮释放事件，当前按下的按钮没有突然变化等）。
  • 处理向用户请求游戏 ID 的对话框。
  • 处理整个游戏的序列化（用于将未完成的游戏加载和保存到磁盘文件；用于处理在 PalmOS 等期望保存状态的平台上的应用程序关闭和重启；用于存储前一个游戏以便在新游戏事件中撤销和重做）。

因此，中端完成了很多工作，这样各个后端就不必担心它。后端所需要做的就是配合确保中端能够正常工作。

中端提供给前端调用的函数 API 在 第4章 中有文档说明。

1.4 杂项实用工具

除了这三个主要结构组件外，Puzzles 代码还包含各种可由上述所有组件使用的实用模块。有一组函数提供平台无关的随机数生成；使内存分配更容易的函数；实现平衡树结构以在复杂算法中必要时使用的函数；以及一些其他杂项函数。所有这些在 第5章 中有文档说明。
